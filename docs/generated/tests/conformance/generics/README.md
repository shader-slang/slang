## Claims

### C-Syntax: Generic parameter list grammar

- **C1**: A generic parameter list is declared with `<...>` after the identifier on a struct, interface, typealias, function, constructor, or subscript operator.
- **C2**: A generic value parameter can be declared with `let N : type` syntax (`generic-value-param-decl`).
- **C3**: A generic value parameter can be declared with the traditional `type N` syntax (`generic-value-param-trad-decl`).
- **C4**: A generic value parameter pack can be declared with `let each N [: type]` syntax.
- **C5**: A generic type parameter can be declared with `[typename] T [: ConstraintType] [= DefaultType]` syntax.
- **C6**: A generic type parameter pack can be declared with `each T [: ConstraintType]` syntax.
- **C7**: A `where` clause can add a conformance constraint (`where T : U`), an equality constraint (`where T == U`), or a coercion constraint (`where U(T)`).

### C-Desc: Description of parameter kinds

- **C8**: A generic value parameter must be of type Boolean, integer, or enumeration.
- **C9**: A generic type parameter pack is a variable-length list of types.
- **C10**: A constraint on a type parameter pack applies to every type in the pack.
- **C11**: Value parameters that are not packs cannot be constrained (only their declared type applies).
- **C12**: A conformance/equality constraint may be declared `optional`; when optional, the expression `T is U` returns `true` when T conforms or equals U, and inside `if (T is U) { ... }` the variable may be used as type U.
- **C13**: A generic value parameter may have a default value via `= init-expr`.
- **C14**: A generic type parameter may have a default type via `= simple-type-spec`.
- **C15**: Type and value parameter packs may be constrained with `where nonempty(P)`, requiring the pack to be non-empty at specialization time.

### C-TypeCheck: Type checking before specialization

- **C16**: Type checking is performed before specialization; an operation on a parameterized type is legal only if legal for all concrete types conforming to the constraints.
- **C17**: A type equality constraint `T == U` causes T to be treated as U for all purposes.
- **C18**: A type conformance constraint `T : U` causes T to be considered to implement all requirements of U.
- **C19**: No assumptions are made about generic value parameters beyond their declared type.
- **C20**: An associated type equality constraint `where T.AssocType == int` requires that T's associated type is exactly int.

### C-Bind: Parameter binding

- **C21**: Arguments can be bound explicitly (angle-bracket syntax after the function/type name).
- **C22**: Arguments can be inferred from the call-site argument expressions (implicit binding).
- **C23**: Mixed binding is allowed: leftmost parameters use explicit arguments, the rest are inferred.
- **C24**: When inference is ambiguous and all inferred types are fundamental scalar or same-length vector types, the type with the highest promotion rank is used (rank table: int8_t … float … double).
- **C25**: When inference yields multiple options for a generic value argument it is an error.
- **C26**: When inference is ambiguous for a non-fundamental type it is an error; explicit binding must be used.

### C-Ex: Examples (normative illustrations)

- **C27**: A generic struct with type and value parameters (`struct Foo<T, let N : uint>`) can be specialized and its fields accessed.
- **C28**: A generic type alias can partially bind a generic type, leaving remaining parameters open.
- **C29**: A generic function taking an `IFunc<T,...>` parameter can be called with a concrete struct implementing `IFunc`.
- **C30**: A generic constructor `__init<T : IInteger>` can initialize a struct from values of type T.
- **C31**: A generic subscript operator `__subscript<T>(T i) where T : IInteger` accepts any IInteger as index.
- **C32**: A type conformance constraint (`where T : IArithmetic`) enables arithmetic operations on T inside the function body.
- **C33**: An optional conformance constraint with `if (T is IInteger)` selects behavior at specialization time.
- **C34**: A `where T.PropertyType == int` equality constraint allows the function to use the associated type as int.
- **C35**: Two generic extensions on the same struct with different type constraints (IFloat vs IInteger) each add their methods only when the constraint is satisfied.
- **C36**: A type parameter pack with `expand each` expands into a comma-separated call sequence.
- **C37**: Multiple function parameter packs must have equal lengths; `expand dotProductHelper(each a, each b, r)` iterates pairwise.
- **C38**: Explicit generic binding (`diffSingle<int32_t>(a, b)`) pins the return type while remaining parameters are inferred.
- **C39**: Both type and value parameters can be inferred from a fixed-size array argument.
- **C40**: Explicit binding resolves an otherwise-ambiguous non-fundamental type inference.

### C-Pack: Pack queries

- **C41**: `__first(P)` and `__last(P)` are partial operations requiring P to be known non-empty (via `where nonempty(P)`).
- **C42**: `__trimFirst(P)` and `__trimLast(P)` are total operations and return an empty pack when applied to an empty pack.
- **C43**: `where nonempty(P)` requires that the pack P is non-empty at specialization time; calling the function with one or more arguments satisfies this constraint.
