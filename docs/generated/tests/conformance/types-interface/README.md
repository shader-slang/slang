# Tests for conformance/types-interface

## Intent

Tests verify interface-type claims in the **language reference** at
[`docs/language-reference/types-interface.md`](../../../../language-reference/types-interface.md).
The bundle exercises: interface declaration syntax and requirement kinds (methods, properties,
constructors, subscripts, call operators, associated types, static const members), conformance
via `struct : IFoo`, interface inheritance, default method implementations, the `override`
modifier, method compatibility rules, visibility rules, generic parameterized interfaces,
interface-typed function parameter specialization, and negative cases for missing requirements
and type restrictions. Functional correctness is verified via `INTERPRET` and `COMPARE_COMPUTE -cpu`;
emission patterns are pinned via `SIMPLE -target hlsl` and `SIMPLE -target glsl`.

## Claims

### §Syntax / §Description — Interface members

- **C1** An interface may declare any number of member function prototypes without a body; a conforming struct must provide compatible implementations with the same names.
- **C2** An interface may declare member functions _with_ a body (default implementations); the conforming struct inherits these without needing to override them.
- **C3** Inheriting types may override a default member function using the `override` modifier.
- **C4** An interface may declare constructor requirements (`__init(...)`); the conforming struct must provide a matching constructor.
- **C5** An interface may declare property requirements with `get;` and/or `set;` without bodies; the conforming struct provides compatible get/set implementations.
- **C6** A property requirement may alternatively be satisfied by declaring a compatible variable with a matching name in the conforming struct.
- **C7** An interface may declare subscript operator requirements without bodies; the conforming struct provides compatible get/set implementations.
- **C8** An interface may declare function call operator requirements; the conforming struct provides a compatible implementation.
- **C9** An interface may declare associated named types (`associatedtype`); a conforming struct satisfies the requirement by providing a `typealias` with the same name.
- **C10** An associated named type requirement may also be satisfied by declaring a nested struct with the same name in the conforming struct.
- **C11** An associated named type may carry a conformance constraint (`associatedtype T : IFoo`); the provided type must conform to that interface.
- **C12** An interface may declare `static const int` data members without initializers; the conforming struct provides matching static const members.
- **C13** An interface may declare `static const bool` data members; the conforming struct provides matching static const bool members.
- **C14** The type of a `static const` interface requirement must be `int` or `bool`; using `float` is rejected with E30306.
- **C15** Interface member functions may be static or non-static.

### §Description — Compatibility rules

- **C16** A member function implementation is compatible with an interface requirement when the impl function can be called with the interface parameter types and the return type can be converted to the interface return type (Remark 1).
- **C17** A member function whose return type is not convertible to the interface return type is rejected with E38106.
- **C18** A struct that declares conformance to an interface but fails to provide a required method is rejected with E38100.
- **C19** A struct that fails to provide a required associated type is rejected with E38100 naming the missing type.
- **C20** A struct that fails to provide a required static const data member is rejected with E38100 naming the missing member.

### §Description — Visibility

- **C21** Interface member visibility may be declared `public` or `internal`; the default is the same as the interface visibility.
- **C22** When a struct implements an interface member, the implementation's visibility may not be _higher_ than the requirement (higher visibility causes an error); it may be lower.

### §Description — Interface inheritance

- **C23** An interface may inherit from another interface using `:` in the declaration; inherited members add to the inheriting interface.
- **C24** A struct conforming to a child interface must satisfy all requirements from both the child and parent interfaces.

### §Remark 2 — Generic interface

- **C25** An interface may be parameterized with generics (`interface IFoo<T>`); a struct conforms by providing a concrete type argument.

### §Interface-Conforming Variants

- **C26** A variable declared with an interface type is an interface-conforming variant; it may hold any conforming type.
- **C27** Interface-typed function parameters do not impose the variant restrictions when invoked with compile-time-known types.
- **C28** When a function with an interface-typed parameter is called with a compile-time-known concrete type, the function is specialized for that concrete type (no dynamic dispatch overhead).

### §Non-Normative (informational only)

- **C29** Memory layout and dispatch mechanism of interface-conforming variants are explicitly unspecified and non-normative.
