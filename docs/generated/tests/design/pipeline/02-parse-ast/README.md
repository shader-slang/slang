# Tests for pipeline/02-parse-ast

## Intent

Tests verify the parser-stage behaviors described in
[`docs/generated/design/pipeline/02-parse-ast.md`](../../../design/pipeline/02-parse-ast.md):
two-stage parsing (decl-stage vs body-stage), syntax-as-declaration
via the `SyntaxParseInfo` table, error recovery, the major AST node
families (`Decl`, `Expr`, `Stmt`, `Type`, `Modifier`), the
`<` disambiguation strategy at the heart of `## Generics ambiguity`,
the generic-constraint spellings the doc enumerates (inheritance
clause, `where` type/equality bounds, `nonempty(...)`,
`countof(...) == IntExpr`, `__hasDiffTypeInfo(...)`, the `optional`
prefix), interface associated-type / `__constraint` requirements,
the checker's builtin-operator rewrite named under
`### The major node families`, modifier and attribute parsing
(including the `[NoDiscard]` example), and the `## Failure modes`
parse-time diagnostics (including the keyword-used-as-name warning).
The bundle deliberately does not duplicate
`syntax-reference/tokens` (lex-surface) or
`syntax-reference/keywords-and-builtins` (keyword spellings); claims
here all hinge on something the parser (or the checker pass the doc
explicitly describes) does after token assembly.

Coverage strategy: at least one functional test for every named
sub-section in the source doc that yields a slangc/slangi-observable
claim. Most tests use `//TEST:INTERPRET` because parser claims are
target-independent — multi-backend repetition would only add runtime
cost; the matrix-operand probe uses CPU `COMPARE_COMPUTE` because the
suite has no interpreter-with-matrix precedent. Failure-mode and
rejection claims use `//DIAGNOSTIC_TEST:SIMPLE` and pin the
diagnostic code (E####/W####) the compiler emits.
