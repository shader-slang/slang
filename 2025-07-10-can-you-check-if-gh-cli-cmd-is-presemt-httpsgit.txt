● The pull_request_review_comment trigger we
  removed was for inline code review comments
   - the ones you make directly on specific
  lines of code in a PR diff.

  Here's the difference between GitHub
  comment types:

  What we kept:
  - issue_comment - Comments on issues OR
  general PR comments (at the bottom)
  - pull_request - When PRs are
  opened/updated

  What we removed:
  - pull_request_review_comment -
  Line-by-line code review comments on
  specific code lines

  Why we removed it:
  1. Reviewer feedback - @csyonghe asked "Do
  we need pull_request_comment?"
  2. Redundant functionality - Most Claude
  interactions happen via general PR
  comments, not inline code reviews
  3. Cleaner workflow - Reduces trigger
  complexity and potential noise

  Example of what we removed:
  # This trigger type was removed:
  pull_request_review_comment:
    types: [created]

  This means Claude won't respond to comments
   like:
  // On line 42 of main.cpp
  @claude this function looks inefficient

  But Claude will still respond to:
  // General PR comment
  @claude can you review this PR and optimize
   the performance?

  The removal makes sense because most AI
  assistant interactions are better suited
  for general discussions rather than
  micro-managing individual code lines.
